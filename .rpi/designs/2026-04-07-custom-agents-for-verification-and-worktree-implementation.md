# Design: Custom Agents for Verification and Worktree Implementation

## Summary

Add two Claude Code custom agents to the RPI toolkit: a **verification agent** that runs spec conformance checks as a read-only subagent, and a **worktree implementation agent** that executes plan phases in an isolated git worktree on a dedicated branch, merging to the base branch on completion. Both are Claude Code-specific enhancements installed by `rpi init`; core logic stays in existing MCP tools.

## Context

The research (.rpi/research/2026-04-07-rpi-improvements-via-claude-code-internals.md, Finding #3) identified custom agents as an untapped Claude Code extension point. Two pain points motivate this:

1. **Verification blocks implementation**: Today, `/rpi-verify` runs in the main conversation — either interrupting implementation or being deferred to the end. A subagent can verify each phase in the background while the next phase starts.

2. **No implementation isolation**: `/rpi-implement` works directly on the current branch. If an implementation goes wrong mid-way, cleanup requires manual `git reset` or `git stash`. A worktree provides atomic, all-or-nothing semantics — the base branch stays clean until everything passes.

Claude Code supports custom agent definitions as markdown files in `.claude/agents/`. Agents can be restricted to specific tools and can run in isolated git worktrees (`isolation: "worktree"` on the Agent tool call).

## Constraints

- **Claude Code only**: Agent definitions are tool-specific. Other MCP clients continue using `rpi_verify_*` tools directly and implementing in-place. The `/rpi-implement` skill must degrade gracefully when agents are unavailable.
- **Existing MCP tools are the brain**: Agents are thin orchestration wrappers that call existing tools (`rpi_verify_spec`, `rpi_verify_completeness`, `rpi_chain`, etc.). No new MCP tools are needed.
- **No inter-agent communication**: The verification agent returns results to the main conversation. It cannot directly signal the implementation agent.

## Components

### 1. Verification Agent (`rpi-verify`)

A read-only subagent that verifies implementation against specs and plans.

**Definition file**: `.claude/agents/rpi-verify.md`

**Tool restrictions**: Read, Glob, Grep, Bash, LSP + all `mcp__rpi__*` tools. No Write, Edit, or NotebookEdit — verification must never modify code.

**Prompt content**: Instructions to resolve the artifact chain from a given plan/spec path, extract scenarios via `rpi_verify_spec`, check each scenario against actual code with file:line references, run `rpi_verify_completeness` and `rpi_verify_markers`, and return a structured summary (pass/fail per scenario, blockers found, overall verdict).

**How it's used**:
- By `/rpi-implement`: spawned after each phase completes (or after all phases) via `Agent(subagent_type: "rpi-verify", prompt: "Verify phase N of <plan-path>")`. Runs in the background while implementation continues.
- Manually: users can spawn it directly for ad-hoc verification without invoking the full `/rpi-verify` skill.

**Relationship to `/rpi-verify` skill**: The skill remains the user-facing entry point (works across all tools, produces a `.rpi/reviews/` report). The agent is the automation primitive — lightweight, returns results inline, no report file. The skill could internally use the agent on Claude Code, but this is optional and not part of this design.

### 2. Worktree Implementation Agent (`rpi-implement-worktree`)

An autonomous agent that implements all plan phases in an isolated git worktree.

**Definition file**: `.claude/agents/rpi-implement-worktree.md`

**Tool restrictions**: All tools (needs Write, Edit, Bash for implementation). MCP tools included.

**Prompt content**: Instructions to implement all remaining plan phases given a pre-resolved context bundle (plan content, spec scenarios, design constraints, list of files to read). The agent:
- Implements each phase following plan instructions
- Runs tests/linters after each phase (success criteria from plan)
- Commits after each passing phase with descriptive messages
- Updates plan checkboxes as phases complete
- Returns a structured summary: phases completed, tests run, commits made, any issues encountered

**Branch naming**: The worktree gets its own branch automatically (Claude Code manages this). The branch name is not controlled by the agent definition — it's generated by Claude Code's worktree system.

**How it's used by `/rpi-implement`**:

```
Main conversation                          Worktree agent
─────────────────                          ──────────────
1. Read plan, resolve chain
2. Pre-review: present all phases
   to user for approval
3. Spawn worktree agent with full    →     4. Receive context bundle
   context bundle                          5. Implement phases sequentially
   (isolation: "worktree")                 6. Run tests after each phase
                                           7. Commit after each phase
                                           8. Return summary + branch info
9. Receive result                    ←
10. Spawn verification agent         →     (reads worktree branch)
11. Receive verification result      ←
12. If all pass + no manual items:
    auto-merge to base branch
13. If manual verification needed:
    pause, show diff, wait for user
14. Merge, update plan → complete
```

**Merge strategy**: Regular merge (not squash) to preserve per-phase commit history. Merge commit message references the plan path. After merge, the worktree branch is cleaned up.

**Fallback when agents unavailable**: The `/rpi-implement` skill detects whether it's running in Claude Code (agents available) or another tool. When agents aren't available, it falls back to the current behavior: in-place implementation on the current branch, no worktree isolation.

### 3. Agent Installation

Agents are embedded in the Go binary alongside skills and installed by `rpi init` / `rpi update`.

**Embedded assets**: `internal/workflow/assets/agents/rpi-verify.md` and `internal/workflow/assets/agents/rpi-implement-worktree.md`

**Installation target**: `.claude/agents/` (Claude target only). The `targetConfig` for Claude gains `"agents"` in its `subdirs` list. `agents-only` and `opencode` targets do not install agent definitions (neither supports Claude Code agents).

**New function**: `workflow.InstallAgents(agentsDir string, force bool)` — mirrors `InstallSkills` but simpler (no frontmatter injection needed, agents are Claude Code-only).

**Sync flow addition**: `syncProject()` calls `InstallAgents` after `InstallSkills` when target is Claude.

### 4. Skill Prompt Updates

The `/rpi-implement` SKILL.md gains a new section on worktree mode:

```markdown
## Worktree Mode (Claude Code)

If the `rpi-implement-worktree` agent is available:
- After pre-review approval, spawn the agent with isolation: "worktree"
- Pass the full context bundle (plan content, spec scenarios, constraints, file list)
- On agent completion, spawn the rpi-verify agent against the worktree branch
- Auto-merge if verification passes and no manual verification items exist
- If manual verification items exist, present the diff and wait for approval
- After merge, update plan status to complete

If agents are not available, implement in-place (current behavior).
```

## Risks

- **Worktree agent autonomy**: The agent implements without per-phase user checkpoints. Mitigated by: (a) pre-review of all phases before spawning, (b) verification after completion, (c) merge approval gate for manual items. If the agent's output is wrong, the base branch is unaffected — discard and retry.
- **Context limits in worktree agent**: Large plans with many files could exceed the agent's context. Mitigated by: the context bundle is curated by the main conversation (only includes what the agent needs, not the entire chain). The agent can also call `rpi_context_essentials` if context is lost.
- **Merge conflicts**: Unlikely for a single worktree merging to an unchanged base branch. If the user made changes to main during implementation, a conflict is possible. Mitigated by: standard git merge conflict resolution; the user can also rebase the worktree branch first.

## Out of Scope

- Plugin packaging (bundles agents + skills + hooks into one installable unit)
- Per-skill hooks (e.g., `/rpi-implement` having its own `PreToolUse` hook)
- Artifact navigator agent (MCP tools suffice)
- OpenCode or agents-only agent definitions (no equivalent system)
- New MCP tools (both agents use existing tools only)
